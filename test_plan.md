# TMAP POI Cluster — 단계별 테스트 가이드

> **목적**: 본 문서는 프로토타입의 모든 요구사항(`raedme.me` 참조)이 의도한 대로 동작하는지를 사람이 브라우저에서 손으로 확인하기 위한 단계별 시나리오를 제공한다. 자동화 E2E는 의도적으로 미포함이며, 추후 Playwright 등으로 그대로 변환할 수 있는 형태로 작성되었다.

## 1. 지도 기본 인터랙션

| 단계 | 동작 | 기대 결과 |
|---|---|---|
| 1.1 | 지도를 마우스로 드래그 | 지도가 따라 움직임. 검색바/툴바는 고정 |
| 1.2 | 마우스 휠/트랙패드로 위/아래로 빠르게 스크롤(연속 burst) | 한 번의 burst가 **단 한 번의 부드러운 줌 단계 변경**으로 통합됨(타일 reload 1회). 노치별로 N단계가 차곡차곡 끊어 보이지 않음 — `useTmapMap`의 `attachDebouncedWheelZoom`이 50ms 윈도로 deltaY를 누적해 한 번만 `setZoom`을 호출 |
| 1.3 | **지도 좌하단**의 커스텀 줌 컨트롤 (`+ / −` 세로 스택) 확인 | (a) TMAP SDK 내장 우측 zoomControl은 **렌더되지 않음** (`zoomControl: false`로 OFF), (b) 좌하단 흰 카드에 위에서부터 `+`와 `−` 두 버튼만 세로로 인접 배치 — **가운데 줌 레벨 숫자 표시는 없음** (spec rollback), (c) 사이드패널이 열려있어도 그 오른쪽(맵 영역의 좌하단)에 위치하므로 가려지지 않음 |
| 1.3a | `+` 버튼 클릭 | 줌 한 단계 증가. zoom 20에 도달하면 `+`가 회색으로 disabled (내부적으로 `zoom_changed`를 구독하여 경계 추적 — UI에 숫자는 안 보이지만 disable 가드는 유효) |
| 1.3b | `−` 버튼 클릭 | 줌 한 단계 감소. zoom 7에 도달하면 `−`가 disabled |
| 1.4 | 모바일 뷰포트(또는 DevTools device toolbar)에서 두 손가락 핀치 | 줌 변화 (Chromium은 핀치를 `wheel` + `ctrlKey`로 보내므로 동일 디바운스 경로로 부드러움) |
| 1.5 | 트랙패드에서 한 번의 강한 플릭 | 한 burst당 최대 4단계까지만 줌 변경 (순간이동 방지 cap) |
| 1.6 | **지도 우하단**의 ◎ `현재 위치로 이동` 버튼 (geolocation 허용 시만 노출) 클릭 | 지도가 `flyToPoi`로 사용자 위치로 부드럽게 pan + zoom 16. **이 system-driven 카메라 변화는 추천목적지 새로고침 버튼을 띄우지 않음** (`flyToPoi`의 setZoom은 `consumeProgrammaticZoom`으로 user-move에서 제외) |
| 1.6a | 위치 권한을 거부한 상태에서 우하단 확인 | ◎ 버튼이 **렌더되지 않음**. 그 자리에 기존 amber 안내 배너("위치 권한이 거부되어...")만 노출됨 — 두 UI가 겹치지 않음 |
| 1.6b | viewer-only 모드(공유 cluster 진입)일 때 좌하단 확인 | "읽기 전용 모드..." 배너가 좌하단 zoom 컨트롤 **위쪽 한 단**(`bottom-16`)에 노출되어 zoom 컨트롤과 겹치지 않음 |

## 2. 사이드패널: 검색 + 검색 결과 + 라벨 핀

| 단계 | 동작 | 기대 결과 |
|---|---|---|
| 2.1 | 사이드패널 상단의 검색창에 `강남역 카페` 입력 후 Enter | (a) 사이드패널에 "검색 결과 (n건)" 섹션 노출, (b) **지도에 빨간 임시 핀들이 등장**, (c) 각 핀 아래에 흰 알약 모양의 **POI 이름 라벨**이 함께 표시됨, (d) **지도 카메라가 자동으로 fit되어 모든 핀이 한 화면에 보임** (`fitBounds` + 18% 여백, max zoom 17) |
| 2.1a | 결과가 1건만 반환되도록 매우 구체적인 키워드(예: 정확한 가게 이름)로 검색 | 카메라가 그 핀 위치로 `setCenter` + `setZoom(17)` 으로 줌인됨 |
| 2.1b | 결과가 매우 가까이 모여있는 키워드(예: 같은 건물 내 여러 매장)로 검색 | over-zoom 없이 max zoom 17에서 cap됨 — 사용자 컨텍스트가 살아있음 |
| 2.1c | 결과가 0건인 키워드로 검색 | 지도 카메라는 그대로 유지(기존 위치 보존). 사이드패널에는 "검색 결과 (0건)" 표시 없음(섹션 자체가 사라짐) |
| 2.2 | 지도를 드래그/줌 | 라벨이 핀과 함께 매끄럽게 이동 (projection 기반 실시간 위치 갱신) |
| 2.3 | 검색 타입 dropdown을 `이름`으로 변경하고 `스타벅스` 검색 | 이름 기반 결과만 반환 |
| 2.4 | 검색 타입을 `주소`로 변경하고 `서울특별시 강남구 테헤란로` 검색 | 주소 기반 결과 반환 |
| 2.4a | 검색 결과 각 행 좌측 | **48×48 사진 썸네일**이 비동기 채워짐. 사진 fetch는 `lib/poiPhoto.ts`의 Gemini 그라운딩 → 회수 실패한 POI는 카테고리 이모지 아바타로 표시되고 행은 결코 가려지지 않음 |
| 2.5 | 사이드패널 검색 결과 행의 **본문(사진+텍스트) 영역** 클릭 | (a) 사이드패널이 **POI 상세화면**으로 push (사이드패널 헤더에 통일된 `←` 뒤로가기 + POI 이름이 표시), (b) 지도가 그 POI로 부드럽게 줌인 (`flyToPoi`, zoom 17) |
| 2.5a | 검색 결과 행 **우측의 ➕ 원형 버튼** 클릭 (메인 사이드패널) | **클러스터 선택 모달(`ClusterPickerModal`)이 즉시 등장**: (a) 사용자 소유 클러스터 목록 + "+ 새 클러스터 만들기" 버튼이 보임, (b) 모달에서 cluster를 고르면 그 cluster에 POI가 추가됨, (c) **행의 ➕ 아이콘은 변하지 않음** — ✓로 바뀌지 않고 그대로 ➕ 유지(spec: "클러스터에 추가되더라도 클러스터 추가 아이콘의 상태는 변하지 않는다"), (d) **상세화면으로 push되지 않음** (➕는 행 본문 클릭과 별개의 stopPropagation 핸들러) |
| 2.5a-2 | 같은 행의 ➕를 **연속해서 다시 클릭** | picker 모달이 또 열림. 같은 클러스터를 또 선택해도 멱등(중복 추가 안 됨, 그 행이 `이미 있음`으로 disabled). 다른 클러스터를 선택하면 그 클러스터에도 추가됨 — 즉 같은 POI를 여러 클러스터에 손쉽게 넣을 수 있음 (sticky한 ✓가 없는 이유) |
| 2.5b | POI 상세화면이 열린 상태에서 헤더 좌측 `←` 클릭 | 사이드패널이 **직전 화면(검색 결과)** 으로 pop. 임시 검색 핀/라벨은 그대로 유지. 검색결과 행의 ➕ 아이콘 상태는 **항상 ➕** (애초에 ✓로 바뀐 적이 없으므로 "리셋"이라는 개념 자체가 불필요) |
| 2.5c | 모달에서 "+ 새 클러스터 만들기" → 이름 입력 → "만들고 추가" | (a) 모달이 닫히고, (b) "내 클러스터" 리스트에 그 이름의 새 클러스터가 등장, (c) localStorage `tmap_clusters_v2` 에 저장되며 `mapCenter`/`mapZoom`은 현재 지도 상태로 스냅샷, (d) 새 클러스터는 자동 활성화되지 않음(사용자가 의도적으로 "내 클러스터"에서 클릭해야 상세뷰로 진입) |
| 2.5d | 같은 POI를 **다른** 클러스터에 또 추가 시도 | 모달의 그 클러스터 행이 `이미 있음` (disabled). 다른 클러스터는 정상 선택 가능 |
| 2.6 | 검색창 우측의 `×` 클릭 | 검색 결과 섹션이 닫히고 추천 목적지가 다시 노출됨. 임시 검색 핀/라벨도 모두 사라짐 |
| 2.8 | 지도 중심을 다른 도시로 드래그 후 다시 검색 | TMAP 검색이 새 지도 중심을 bias로 호출되는지 DevTools Network 탭에서 `centerLat`/`centerLon` 파라미터로 확인 |

## 7. 클러스터 저장 / 클러스터 상세뷰 / 좋아요 / 노트

| 단계 | 동작 | 기대 결과 |
|---|---|---|
| 7.1 | 메인 사이드패널의 추천 목적지에서 ➕ 버튼 클릭 | **picker 모달이 등장** — 메인 검색결과 / POI 상세화면의 ➕ 와 동일한 흐름. 모달에서 기존 cluster를 선택하거나 새 cluster를 만들 수 있음 |
| 7.1a | 메인 사이드패널의 검색결과 또는 추천에서 ➕ → picker 모달 → "+ 새 클러스터 만들기" → 이름 `주말 강남` | 새 클러스터가 즉시 생성됨. 그 클러스터를 "내 클러스터"에서 클릭하면 클러스터 상세뷰가 열리고 그 POI가 첫 목적지로 보임 |
| 7.2 | 우상단 영역 확인 | 우상단 툴바에는 `클러스터 저장` / `공유` / `초기화` 버튼이 **존재하지 않음** (spec: "오른쪽 상단에 클러스터 저장 공유 초기화 버튼은 없앤다."). 활성 클러스터가 있으면 `클러스터: 이름` 라벨, viewer-only 모드면 `내 계정에 복사` 버튼, Google login 버튼만 보임 |
| 7.3 | (구) 이름 비워두고 저장 시도 | **삭제됨** — 저장 모달 자체가 사라졌으므로 이 시나리오 무효 |
| 7.4 | 새 클러스터 생성은 picker 모달의 `+ 새 클러스터 만들기` → 이름 `주말 강남` 입력 → 만들기 | picker가 닫히고 cluster가 즉시 storage(`tmap_clusters_v2`)에 저장됨. "내 클러스터" 리스트에 노출. **현재 활성 cluster로 자동 전환되지는 않음** — 사용자가 "내 클러스터"에서 클릭해야 상세뷰 진입 |
| 7.5 | DevTools → Local Storage → `tmap_clusters_v2` | 저장된 객체에 `pois[]`, `feedback{}`(처음에는 `{}`), `mapCenter`, `mapZoom`, `ownerId`, `ownerName`, `createdAt`, `updatedAt`가 포함됨. v1 키는 더 이상 사용되지 않음 |
| 7.6 | 사이드패널의 **"내 클러스터"** 인라인 섹션 확인 | 방금 저장한 항목이 자동 노출. `n개 목적지 · 📝0 · ❤️0` 라인. **"현재" 배지는 더 이상 노출되지 않음** (활성 클러스터 컨텍스트는 클러스터 상세뷰가 열려있는 동안만 의미가 있고 ← 뒤로가면 즉시 정리되므로) |
| 7.7 | "내 클러스터" 항목을 클릭 | (a) 사이드패널이 **클러스터 상세뷰**로 전환됨 (좌상단 `←` 뒤로가기 + 클러스터 이름 + 목적지 카운트 헤더), (b) 지도가 fitBounds로 모든 핀이 보이도록 자동 줌, (c) 추천/내 클러스터 섹션은 사라지고, (d) **클러스터 상세뷰 상단에는 검색창이 그대로 노출**되어 그 클러스터에 추가할 다른 POI를 검색할 수 있음 |
| 7.7a | 좌상단 `←` 클릭 | 사이드패널 history stack을 한 단계 pop → 메인 뷰(검색/즐겨찾기/내 클러스터/추천)로 복귀. **활성 클러스터의 핀이 일제히 사라짐** (`goBackInPanel` 이 cluster pop을 감지하면 `pois`/`currentClusterId`/`feedback`/`searchResults`를 한 번에 reset). 같은 cluster를 다시 보려면 "내 클러스터"에서 다시 클릭 |
| 7.7b | 클러스터 상세뷰 안에서 검색창에 키워드 입력 후 검색 | (a) 검색결과가 클러스터 상세뷰 내부 상단에 인라인 표시, (b) 임시 검색 핀이 지도 위에 등장하며 **이미 있는 클러스터 핀과 함께** 동시 노출 (둘 다 보임), (c) 결과 행 우측의 ➕ 클릭 시 **picker 없이 그 클러스터에 즉시 추가** — 새 빨간 클러스터 핀이 등장함과 동시에 그 POI의 임시 검색 핀은 사라짐. **➕ 아이콘 자체는 ✓로 변하지 않고 그대로 ➕ 유지** (spec: "클러스터에 추가되더라도 클러스터 추가 아이콘의 상태는 변하지 않는다") |
| 7.7c | 7.7b의 추가 직후 ← 클릭 | 메인 뷰로 복귀하면서 클러스터 핀과 검색 핀이 모두 사라짐. 다시 같은 cluster를 열면 **방금 추가한 POI가 목적지 목록에 영구 보존** (storage `tmap_clusters_v2`의 그 cluster `pois`에 `updateCluster()`로 즉시 영속화되었음) |
| 7.8 | 클러스터 상세뷰의 목적지 행 우측 | 행 우측에 `📝 0` (또는 노트수) 와 `❤️/🤍 0` (또는 좋아요수) 카운트가 항상 보임. 검색결과/추천 행에는 이 카운트가 **절대** 보이지 않음 (해당 카운트는 클러스터 상세뷰 전용) |

| 7.10 | 펼친 영역의 사진(있으면)과 이름/주소 | `<PoiPhoto>` 컴포넌트로 비동기 채워짐 (§ 14.4.2와 동일 캐시 공유) |

| 7.12 | Google 로그인 → 다시 좋아요 클릭 | 버튼이 `❤️ 좋아요 취소`로 토글되고 우측 ❤️ 카운트가 +1, 행 우측 카운트도 즉시 갱신, 우측 작은 글씨로 본인 계정명이 노출 |
| 7.13 | DevTools Local Storage `tmap_clusters_v2` → 해당 클러스터의 `feedback["<poi.id>"].likes` | `[ { userId, userName, ts } ]` 1건이 즉시 영속화. (mutation → 즉시 `updateCluster()` 호출 검증) |
| 7.14 | 노트 textarea에 `여기 진짜 좋다`라고 입력 후 `추가` | 노트 패널 목록에 본인 계정명 + 시각 + 본문이 새 yellow 박스로 추가. `📝 카운트` +1 |

| 7.16 | 본인 노트 우하단의 `삭제` 클릭 | 노트가 즉시 사라지고 `📝` 카운트가 −1 |
| 7.17 | "내 클러스터" 인라인 섹션의 `삭제` 클릭 | 항목 제거됨. 모든 클러스터가 사라지면 **"내 클러스터" 섹션 자체가 자동 숨김** |

## 9. 공유받은 사람의 시점 (View-only + 좋아요/노트 + 복사)

| 단계 | 동작 | 기대 결과 |
|---|---|---|
| 9.1 | 시크릿 창에서 공유 URL로 진입 | 사이드패널 메인 뷰의 ➕ 버튼이 사라지고 (목적지 편집 불가), 클러스터 상세 카드의 `제거` 버튼도 보이지 않음. 검색바 자체는 사용 가능(다른 검색을 할 수는 있음) |
| 9.1a | 공유받은 클러스터의 목적지 펼침 → `🤍 좋아요` 또는 노트 입력 시도 | (a) 비로그인이면 alert로 로그인 요청, (b) 로그인 후엔 정상 동작. 입력 결과는 즉시 localStorage에 영속화되며 같은 브라우저의 다른 탭에서 새로고침하면 그 좋아요/노트가 보임 (실제 백엔드 연결 시 다른 사용자에게도 즉시 broadcast됨을 시뮬레이션) |
| 9.1b | 공유 URL을 받은 사람의 좋아요 → 원 owner의 사이드패널 "내 클러스터" 행 카운트 | `❤️ n`이 +1로 반영됨 (같은 브라우저 모델 한정. 백엔드 연결 시 owner의 다른 디바이스도 동일하게 갱신) |
| 9.1c | **공유 URL 진입 즉시** 사이드패널 좌상단 ← 로 메인으로 복귀 | 메인 뷰 "내 클러스터" 섹션에 그 클러스터가 **`👥 공유받음` 뱃지와 함께 즉시 노출**. 비로그인이어도 동일. (`App.tsx` hash 핸들러가 viewer의 localStorage에 cluster를 seed 함) |
| 9.1d | 9.1c의 viewer가 그 행의 `내 목록에서 제거` 클릭 | 자기 사이드패널 리스트에서만 사라짐. (mock 한정: localStorage record도 같이 지워지지만, 실제 백엔드에선 viewer-only unfollow API로 매핑되어 원본은 유지) |
| 9.1e | viewer가 9.1d로 제거한 후 같은 공유 URL을 다시 열기 | 다시 자동 seed되어 리스트에 즉시 노출되고 정상 동작 |
| 9.1f | viewer가 클러스터 상세뷰에서 목적지 행 우측의 `제거` 버튼을 찾으려 함 | **버튼 자체가 보이지 않음** — `viewOnly = !ownedByMe`로 자동 잠금. 좋아요/노트는 여전히 가능 (spec: 목적지 삭제는 owner만) |
| 9.2 | 우상단 `Google 로그인 (mock)` 클릭 | 버튼이 `Demo User · 로그아웃`으로 변경 |
| 9.3 | 우상단 `내 계정에 복사` 클릭 | 새 hash로 사본이 생성되어 URL이 변경되고, view-only 배지가 사라지며 편집 가능 상태로 전환. 사본의 `feedback`은 빈 상태(`{}`) — 원본의 좋아요/노트는 가져오지 않음 (타인의 흔적이 자기 클러스터에 따라 들어가지 않도록) |
| 9.4 | 로그인 없이 9.3 시도 | "로그인이 필요합니다" alert 표시 |
| 9.5 | **owner 시점**: 자기가 만든 클러스터의 행 우측 | `삭제` 라벨, 클릭 시 영구 삭제(원본 제거). `👥 공유받음` 뱃지는 보이지 않음 |
| 9.6 | **viewer 시점**: 같은 클러스터를 다른 user로 로그인 후 같은 브라우저에서 열기 | 행 우측이 `내 목록에서 제거`로 라벨링되고 `👥 공유받음` 뱃지 노출. owner의 user.id와 다르므로 `isClusterOwnedByMe`가 false |

| 14a.7 | `+ 클러스터에 추가` 클릭 | **클러스터 선택 모달(`ClusterPickerModal`)이 등장** — 사용자 소유 클러스터 목록 + "+ 새 클러스터 만들기" 버튼. POI 상세화면은 어떤 컨텍스트에서 진입했는지 알 수 없으므로 implicit target이 없고 항상 picker가 떠야 함 |
| 14a.7a | 모달에서 기존 클러스터 선택 | 모달이 닫히고 그 클러스터에 POI가 추가됨(localStorage 영속화). 그 클러스터를 "내 클러스터"에서 클릭해 상세뷰로 들어가면 그 POI가 목록에 있음. POI 상세화면의 `+ 클러스터에 추가` 버튼은 그대로 `+`로 남음(모달이 즉시 닫혀 row state 개념이 적용되지 않음) |
| 14a.7b | 모달에서 "+ 새 클러스터 만들기" → 이름 입력 → "만들고 추가" | 새 클러스터가 그 POI 1개로 즉시 생성되며 "내 클러스터"에 노출됨 |
| 14a.8 | View-only 모드(공유 링크로 진입한 viewer)에서 같은 POI 상세화면을 열어 본다면 | `+ 클러스터에 추가` 버튼이 정상적으로 picker를 띄움 — viewer는 자기 소유 클러스터에는 추가 가능 (공유받은 클러스터에는 picker가 자기 소유 cluster만 보여주므로 자동 필터됨) |
| 14a.9 | 상세화면 진입 후 지도 확인 | 진입 시점에 그 POI의 좌표로 부드럽게 줌인됨 (`flyToPoi`, ~380ms) |
| 14a.10 | 같은 POI 상세화면을 두 번 연속으로 push 시도 (검색 결과 행을 연달아 빠르게 두 번 클릭) | history stack이 dedupe되어 한 번만 push됨. ← 한 번이면 메인으로 복귀 |

## 14b. 즐겨찾는 목적지 — 엔트리포인트 + FavoritesView

| 단계 | 동작 | 기대 결과 |
|---|---|---|
| 14b.1 | 즐겨찾기를 한 번도 한 적이 없는 상태 | 사이드패널 메인 뷰의 "내 클러스터" 섹션 **아래**에 한 줄짜리 카드(`⭐ 즐겨찾는 목적지` + `아직 없음 · POI 상세화면에서 ☆를 눌러 추가하세요` + `→`)가 보임. (전체 리스트가 인라인되지 않음) |
| 14b.2 | 14a.5처럼 POI 상세화면에서 즐겨찾기에 1건 추가 후 ← 로 메인 복귀 | 엔트리포인트 카드의 카운트가 `1`로 갱신되고 보조 텍스트가 `1개 저장됨 · 탭해서 전체 보기`로 바뀜 + 우측에 노란 `1` 뱃지 |
| 14b.3 | 엔트리포인트 카드 클릭 | 사이드패널이 **`FavoritesView`로 push**되고 헤더에 통일된 `←` + `즐겨찾는 목적지` 타이틀이 보임. 본문에 즐겨찾기 카드 1개가 사진+이름+주소+× 형식으로 노출 |
| 14b.4 | `FavoritesView`의 카드 본문 클릭 | 검색결과/추천과 동일하게 사이드패널이 **POI 상세화면**으로 push (history stack: main → favorites → poiDetail). ←를 누르면 favorites 리스트로, 한 번 더 누르면 메인으로 복귀 |
| 14b.5 | `FavoritesView`의 카드 우측 `×` 클릭 | 즉시 그 카드만 사라지고 localStorage에서도 제거. 0건이 되면 ⭐ 큰 아이콘과 함께 안내 빈 상태가 노출 |
| 14b.6 | `FavoritesView`에서 헤더 `←` 클릭 | 사이드패널 history stack pop → 메인 뷰로 복귀. 엔트리포인트 카드의 카운트 즉시 갱신 |
| 14b.7 | 같은 POI를 두 번 즐겨찾기 추가 시도 | 중복 항목이 생기지 않고 timestamp만 갱신되어 목록 최상단으로 올라옴(최근 즐겨찾기 순 정렬) |
| 14b.8 | 즐겨찾기를 5건 이상 추가 후 페이지 새로고침 | 모든 항목이 그대로 복원됨 (localStorage 영속화 확인) |

## 부록 A — 알려진 제한 / 의도된 mock 동작

- Google 로그인은 mock입니다. `Demo User`로 표시됩니다. (`raedme.me` § Partially Implemented A)
- 클러스터/좋아요/노트/Gemini 답변은 모두 `localStorage`에 저장됩니다 (§ B). 스키마는 `tmap_clusters_v2` — 옛 v1 데이터는 무시합니다.
- **공유 의미론**: 좋아요/노트는 같은 클러스터를 보는 모든 사용자에게 공유되어야 한다는 스펙이지만, 백엔드가 없으므로 현재는 **같은 브라우저 안에서만** 다른 탭/세션 간 공유됩니다. 실제 백엔드 연결 시 `lib/storage.ts`의 `updateCluster()`만 REST/WebSocket로 교체하면 자동으로 다중 사용자 broadcast가 동작합니다.
- **Gemini는 실제 Google AI Studio API(`gemini-2.5-flash-lite`)를 직접 호출**합니다. 키는 프론트 번들에 포함되므로 운영 시 백엔드 프록시로 이전 권장 (§ C)
- POI 사진은 **Gemini Google Search grounding**으로 회수합니다. 모델이 직접 이미지 URL을 찾지 못하면(혹은 hot-link 차단되어 `<img>`가 onError를 발사하면) 카테고리 이모지 아바타로 graceful fallback (§ D). 사진 호출은 추천 / 검색결과 / 클러스터 detail 카드가 동일한 캐시를 공유합니다.
- **Sticky Note 모델은 폐기**되었습니다. 지도 위에는 핀만 있고, 모든 POI 상세/메모(=노트)/AI는 사이드패널의 클러스터 상세뷰 안에서 렌더링됩니다.

## 부록 B — 디버깅 팁

- `useProjectionLine`이 픽셀을 반환하지 않으면 콘솔에서 `window.Tmapv2`의 존재 여부를 먼저 확인. (현재 사용처는 `PinLabelLayer` 라벨 위치 한정)

- 좋아요/노트가 다른 탭에 반영되지 않으면 storage 이벤트 구독이 없기 때문 — 새로고침이 필요. 백엔드 연결 시 WebSocket/SSE로 push 받으면 해결.
